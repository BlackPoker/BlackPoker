# RST・YAMLドキュメント記述スタイルガイド

このスキルは、BlackPokerプロジェクト内の `.rst` ファイルや `act.yaml` などの `.yaml` ファイルを編集または作成する際に、記述スタイルや用語の一貫性を維持・チェックするためのものです。

## 表記・用語の統一ルール

### 1. 指示語としての「以下」の制限
- **原則**: 後続の文章、表、画像、手順などを指し示す場合は「以下」ではなく **「次」** を使用します。
  - **非推奨（例）**: 「以下の手順に従って選択してください」「以下の表をご覧ください」「以下に、段階ごとの流れを示します」
  - **推奨（例）**: 「次の手順に従って選択してください」「次の表をご覧ください」「次に、段階ごとの流れを示します」
- **例外**: 数値や数量の範囲を示す不等号としての「以下」は、変更せずにそのまま使用してください。
  - **例外例**: 「0以下」「ライフ2以下」「10枚以下」

### 2. 数字表記の統一（半角数字への統一）
- **原則**: 数字（0-9）は原則としてすべて **半角数字** を使用します。全角数字（０-９）は使用しません。
  - **非推奨（例）**: 「１回」「第２引数」「１００ミリ秒」
  - **推奨（例）**: 「1回」「第2引数」「100ミリ秒」
- **例外**: 固有名詞や、全角で記述する必要がある明確な理由がある場合を除きます。

### 3. YAMLにおける更新バージョン（update）のカンマ区切りルール
- **原則**: YAMLファイル（`act.yaml`、`act-frame.yaml`など）の更新バージョンを示す `update` フィールドにおいて、複数のバージョンをカンマで区切る場合、カンマの後に必ず **半角スペース** を1つ入れます。
  - **非推奨（例）**: `update: v8.0,v8.1,v9.0`
  - **推奨（例）**: `update: v8.0, v8.1, v9.0`

### 4. 定義リストにおける見出し直後の改行制限（インデントの伸び防止）
- **原則**: RSTファイルにおいて、定義リスト（太字の見出しの直後にインデントされた詳細を記述する形式）を使用する際、見出し行とインデントされた詳細行の間に **空行（改行）を入れない** ようにします。
- **理由**: 空行を入れてしまうと、RSTの構文解釈において「定義リスト」ではなく「ブロッククォート（引用）」として処理されます。これにより、ビルド時に余分な余白が適用され、インデントが意図せず広がってしまいます。
  - **非推奨（インデントが広くなる例）**:
    ```rst
    **内容**

       アクションの対象は必ず一つになるように設計する。
    ```
  - **推奨（意図通りのインデントになる例）**:
    ```rst
    **内容**
       アクションの対象は必ず一つになるように設計する。
    ```

### 5. 定義説明内での箇条書きリストのネスト禁止（フラットな太字リストへの統一）
- **原則**: 定義リストや説明文の中で、さらに `-` や `*` を使った箇条書きリストを二重にネストして記述することは避けます。
- **理由**: PDFビルド時にリスト環境が二重にネストされ、不自然な縦の隙間や極端に深いインデントが発生するためです。
- **対策**: ネストを避け、`* **[項目]**: 説明文` のように、コロンで区切る形のフラットな太字箇条書きに統一します。
  - **非推奨（隙間ができインデントが深くなる例）**:
    ```rst
    **[1] 全ての (all)**
       - ゲームの場全体...
    ```
  - **推奨（フラットな太字箇割書きの例）**:
    ```rst
    * **[1] 全ての (all)**:
      ゲームの場全体...
    ```

### 6. カギカッコの表記制限（半角化の禁止）
- **原則**: 全角のカギカッコ（「」）は、半角のカギカッコ（｢｣）に **絶対に変換しない**（全角のまま維持する）ようにします。
- **理由**: LaTeXによるPDFビルド時に、半角カギカッコ（｢ ｣）は unicode 処理エラー（`inputenc Error: Unicode character`）を誘発し、ビルドを失敗させる原因になります。もしファイル内に半角カギカッコを見つけた場合は、全角に修正してください。
  - **非推奨（ビルドエラーの原因）**: ｢カードを集めること｣
  - **推奨**: 「カードを集めること」

### 7. 括弧表記 of 統一（半角括弧への統一）
- **原則**: 日本語の地の文中であっても、丸括弧は原則としてすべて **半角の `(` と `)`** を使用します。全角の `（` と `）` は使用しません。
  - **非推奨（例）**: 「引き分け（両プレイヤーが獲得）」
  - **推奨（例）**: 「引き分け(両プレイヤーが獲得)」

### 8. 比較記号の統一（半角記号への統一）
- **原則**: 不等号などの比較記号は、原則として **半角の `>` や `<`** を使用します。全角の `＞` や `＜` は使用しません。
  - **非推奨（例）**: 「ライト ＞ スタンダード ＞ プロ」
  - **推奨（例）**: 「ライト > スタンダード > プロ」

## チェック手順
`.rst` および `.yaml` ファイルを作成または修正する際は、必ず本スキルに定義されたルールと照らし合わせ、不適切な「以下」の表現、全角数字、`update` フィールドのカンマ後のスペース不足、定義リスト内の不要な空行、定義説明内でのリストのネスト、不適切な半角カギカッコ、全角の括弧（`（ ）`）や全角不等号（`＞ ＜`）が含まれていないかをチェック・修正してください。
