# RST・YAMLドキュメント記述スタイルガイド

このスキルは、BlackPokerプロジェクト内の `.rst` ファイルや `act.yaml` などの `.yaml` ファイルを編集または作成する際に、記述スタイルや用語の一貫性を維持・チェックするためのものです。

## 表記・用語の統一ルール

### 1. 指示語としての「以下」の制限
- **原則**: 後続の文章、表、画像、手順などを指し示す場合は「以下」ではなく **「次」** を使用します。
  - **非推奨（例）**: 「以下の手順に従って選択してください」「以下の表をご覧ください」「以下に、段階ごとの流れを示します」
  - **推奨（例）**: 「次の手順に従って選択してください」「次の表をご覧ください」「次に、段階ごとの流れを示します」
- **例外**: 数値や数量の範囲を示す不等号としての「以下」は、変更せずにそのまま使用してください。
  - **例外例**: 「0以下」「ライフ2以下」「10枚以下」

### 2. 数字表記の統一（半角数字への統一）
- **原則**: 数字（0-9）は原則としてすべて **半角数字** を使用します。全角数字（０-９）は使用しません。
  - **非推奨（例）**: 「１回」「第２引数」「１００ミリ秒」
  - **推奨（例）**: 「1回」「第2引数」「100ミリ秒」
- **例外**: 固有名詞や、全角で記述する必要がある明確な理由がある場合を除きます。

### 3. YAMLにおける更新バージョン（update）のカンマ区切りルール
- **原則**: YAMLファイル（`act.yaml`、`act-frame.yaml`など）の更新バージョンを示す `update` フィールドにおいて、複数のバージョンをカンマで区切る場合、カンマの後に必ず **半角スペース** を1つ入れます。
  - **非推奨（例）**: `update: v8.0,v8.1,v9.0`
  - **推奨（例）**: `update: v8.0, v8.1, v9.0`

### 4. 定義リストにおける見出し直後の改行制限（インデントの伸び防止）
- **原則**: RSTファイルにおいて、定義リスト（太字の見出しの直後にインデントされた詳細を記述する形式）を使用する際、見出し行とインデントされた詳細行の間に **空行（改行）を入れない** ようにします。
- **理由**: 空行を入れてしまうと、RSTの構文解釈において「定義リスト」ではなく「ブロッククォート（引用）」として処理されます。これにより、ビルド時に余分な余白が適用され、インデントが意図せず広がってしまいます。
  - **非推奨（インデントが広くなる例）**:
    ```rst
    **内容**

       アクションの対象は必ず一つになるように設計する。
    ```
  - **推奨（意図通りのインデントになる例）**:
    ```rst
    **内容**
       アクションの対象は必ず一つになるように設計する。
    ```

## チェック手順
`.rst` および `.yaml` ファイルを作成または修正する際は、必ず本スキルに定義されたルールと照らし合わせ、不適切な「以下」の表現、全角数字、`update` フィールドのカンマ後のスペース不足、および定義リスト内の不要な空行が含まれていないかをチェックしてください。
