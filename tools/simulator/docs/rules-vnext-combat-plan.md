# 新ルールYAML DSL 戦闘モデル設計・実装計画

本ドキュメントは、BlackPoker 新ルールYAML DSLシミュレーターにおける戦闘モデル（アタック、ブロック、ダメージ判定、および戦闘に伴う状態更新や他システムとの連携）の設計を整理し、今後の段階的な実装に向けた計画・ロードマップを定義するものです。

---

## 1. 設計方針と基本原則

- **戦闘を「フェーズ」ではなく「アクション」として扱う**
  BlackPoker には公式ルール上「戦闘フェーズ」や「バトルフェーズ」といったフェーズ概念は存在しません。すべての戦闘プロセスは、タイミング制約と効果を持つ「アクション」として定義され、ステージ（LIFOスタック）を通じて順次解決されます。
- **GameStateの健全性の維持 (フェーズの再導入禁止)**
  既存のターン＆チャンスモデル（手番・チャンス管理）をそのまま適用し、`phase` 概念は再導入しません。
- **Stage / ActionRequest を用いた戦闘状態の表現**
  一時的な戦闘状態（アタッカーやブロッカーの指定情報）は、可能な限り `ActionRequest` のコンテキストや GameState の軽量な一時オブジェクト（`state.combat`）で表現し、不要な公式外概念を増やさないようにします。
- **エンドアクションにおけるフォグのクリーンアップ前提の維持**
  戦闘の解決やそれによる墓地移動の後、ターン終了は `action.end` アクションとして実行され、フォグはエンドアクションの解決時に全プレイヤー分が各々の墓地へ移動する前提を維持します。

## 2. 旧定義における戦闘関連要素の整理

第9.0版公式ルール（`act.yaml` / `act-frame.yaml`）における戦闘要素は以下の通り定義されています。

### 2.1. アタック (`attack`)
- **旧ID**: `attack`
- **タイプ**: 基本 / 直接アクション
- **速度 (`actSpeed`)**: 通常
- **タイミング (`actTime`)**: メイン
- **使用条件**: プレイヤーは1ターンに1回までこのアクションを起こすことができる。
- **効果内容**:
  - 対戦相手を攻撃するアタッカーをドライブ（`drive`）して指定する。
  - アタッカーは `<攻撃>` ラベルを持つキャラクターのみ指定できる（複数指定可能、ドライブ状態のものは不可）。
  - このターンに場に出たキャラクターは指定できない（`<速攻>` を持つ場合は例外）。

### 2.2. ブロック (`block`)
- **旧ID**: `block`
- **タイプ**: 基本 / 誘発アクション
- **速度 (`actSpeed`)**: 通常
- **タイミング (`actTime`)**: メイン
- **誘発条件**: ターンプレイヤーであり、かつ、アタックアクションが解決した時にアタッカーが1体以上いる場合に自動的に誘発する。
- **効果内容**:
  - 対戦相手（防御側）は、アタッカー毎にそれをブロックするキャラクター（ブロッカー）を指定する。
  - ブロッカーは `<防御>` ラベルを持つキャラクターのみ指定できる（ドライブ状態のものは不可）。
  - 兵士でブロックする場合、1アタッカーに対して複数の兵士を指定できる。

### 2.3. ダメージ判定 (`damageJudge`)
- **旧ID**: `damageJudge`
- **タイプ**: 基本 / 誘発アクション
- **速度 (`actSpeed`)**: 通常
- **タイミング (`actTime`)**: メイン
- **誘発条件**: ターンプレイヤーであり、かつ、ブロックアクションが解決した時に自動的に誘発する。
- **効果内容**:
  - アタッカーとブロッカーを比較し、以下の処理を実行する。
    1. **兵士（アタッカー） vs 兵士（ブロッカー）**:
       - 互いのサイズ（数字）を比較し、小さい方を墓地（`grave`）に移す。
       - サイズが同じ場合は両方を墓地に移す。
       - 1体のアタッカーに対し複数のブロッカーがいる場合、ブロッカーの合計サイズと比較する。
    2. **兵士（アタッカー） vs 防壁（ブロッカー）**:
       - 防壁を表（`face: up`）にする。
       - 防壁が「Joker」であるか、防壁のカードに記載されている数字と同じ数字がアタッカーのカードに含まれている場合、アタッカーを墓地に移す。
       - その後、防壁を墓地に移す。
    3. **ブロックされなかったアタッカー**:
       - アタッカーのサイズ（数字）に等しいダメージを対戦相手プレイヤーに与える（`dealDamage` コマンドの実行）。

## 3. 戦闘状態モデルと解決フロー

過剰な状態オブジェクトを避けるため、戦闘中の状態は GameState 内に軽量な一時オブジェクト `state.combat` を定義して追跡します。

### 3.1. 戦闘状態モデル (`state.combat`)
```typescript
interface CombatState {
  attackerIds: string[]; // アタッカーに指定されたユニットのIDリスト
  blocks: {
    attackerId: string;
    blockerIds: string[]; // ブロッカー（兵士または防壁）のユニットIDリスト
  }[];
}
```

### 3.2. 戦闘の解決フロー（ステージ上の連鎖）

1. **アタックの宣言**
   - 手番プレイヤーが `action.attack` をリクエスト（ステージに積載）。
   - クイックアクションによる割り込みの解決。
   - `action.attack` の解決：
     - 対象ユニットをドライブ状態へトグル。
     - `state.combat` を初期化し、`attackerIds` に指定ユニットを格納。
     - **アタック解決の誘発**: システムが自動的に `action.block` リクエストを作成してステージに積む。

2. **ブロックの宣言**
   - 防御側プレイヤーが `action.block` を解決。
   - クイックアクションによる割り込みの解決。
   - `action.block` の解決：
     - 防御側がアタッカー毎にブロッカーを指定し、`state.combat.blocks` にバインド。
     - **ブロック解決の誘発**: システムが自動的に `action.damageJudge` リクエストを作成してステージに積む。

3. **ダメージ判定の実行**
   - クイックアクションによる割り込みの解決。
   - `action.damageJudge` の解決：
     - `state.combat` に基づき、サイズ比較・防壁特殊判定・プレイヤーへのダメージ（`dealDamage`）を実行。
     - 生じたカード移動（`field -> grave`）により、Legacy Card があれば「世代交代（`nextGeneration`）」が誘発してステージに積まれる。
     - 戦闘終了後、`state.combat` はクリアされる。

## 4. ターン＆チャンスモデルおよび既存アクションとの連携設計

### 4.1. ターン＆チャンスモデルとの関係
- **アタックの実行制限**:
  - `action.attack` は `timing: main` であり、手番プレイヤーかつチャンスプレイヤーであり、ステージが空のときのみリクエスト可能。
- **ブロックとダメージ判定の誘発**:
  - これらは直接リクエストされるアクションではなく、先行アクションの解決時に自動で誘発されてステージに積まれる「誘発アクション」として整理します。
  - チャンスプレイヤーが誰であるかに関わらず、ステージに積まれた誘発リクエストは順番に自動解決されます（ただし、解決前にクイックアクションを割り込むチャンスは各プレイヤーに与えられます）。

### 4.2. 既存アクションとの相互作用
- **アップ / ダウン**:
  - アタック解決後、ダメージ判定解決までの間に `action.up` や `action.down` をアタッカーやブロッカーに適用することで、戦闘結果（生死や被ダメージ量）を動的にコントロール可能です。
- **ツイスト (`action.twist`)**:
  - アタッカーやブロッカーの指定条件は「チャージ状態であること」です。
  - 攻撃宣言前に対象をツイストでチャージに戻すことで戦闘に参加可能になります。
  - *仕様確認が必要な点*: アタック宣言後、ダメージ判定解決までの間にアタッカーがツイスト等によってチャージ状態に戻された場合、アタッカー指定が解除されるのか、あるいはドライブ状態でなくてもダメージ判定はそのまま行われるのか、ルールの厳密な定義を今後整理する必要があります。
- **カウンター (`action.counter`)**:
  - ステージ上の `action.attack` や `action.block` リクエスト自体を対象としてカウンターし、戦闘プロセスそのものを途中で取り消すことが可能です。
- **要塞 (`trump.fortress`)**:
  - 要塞はアタックをブロックするための「キャラクター」ではないため、戦闘ダメージを直接軽減・無効化することはありません（投擲などの効果によるダメージのみが対象）。
- **防壁**:
  - 表向きの防壁はキャラクター（防御ラベル所持）としてブロックに参加可能。
  - 裏向きの防壁をブロッカーに指定した場合、ダメージ判定解決時に表にし、防壁特有の即死判定（Joker判定、数字一致判定）を実行したのち墓地に送られます。
- **世代交代 (`action.nextGeneration`)**:
  - 戦闘ダメージ判定の結果、アタッカーやブロッカー（キャラクター・防壁）が場から墓地（`field -> grave`）へ移動した際、それらを構成するカードが Legacy Card である場合、正常に世代交代が誘発・実行されます。

## 5. 今後の実装フェーズ・ロードマップ

戦闘モデルの再現に向け、以下の5つの段階に分割して実装を進めます。

### Phase 14.1: アタックアクションの最小実装
- **目的**: アタッカーの指定、ドライブ状態へのトグル、および `state.combat.attackerIds` への記録を実証する。
- **やること**:
  - `action.attack` のYAML定義の追加。
  - アタッカー指定条件（攻撃ラベル、チャージ状態、速攻/ターン制限）のバリデーション実装。

### Phase 14.2: ブロックアクションの最小実装
- **目的**: 防御側によるブロッカーの指定、および `state.combat.blocks` への記録を実証する。
- **やること**:
  - `action.block` のYAML定義の追加。
  - ブロッカー指定条件（防御ラベル、チャージ状態）のバリデーション実装。
  - アタック解決時にブロックアクションが自動誘発してステージに積まれる仕組みの構築。

### Phase 14.3: ダメージ判定アクションの最小実装 (基本戦闘)
- **目的**: 兵士同士の基本的なサイズ比較による戦闘結果の解決を実証する。
- **やること**:
  - `action.damageJudge` のYAML定義の追加。
  - ブロック解決時にダメージ判定が自動誘発してステージに積まれる仕組みの構築。
  - 兵士 vs 兵士のサイズ比較、敗北側の墓地送り（`moveToGraveyard`）、およびブロックされなかった際のスルーダメージ（`dealDamage`）の実装。

### Phase 14.4: 戦闘による防壁判定と世代交代連携
- **目的**: 防壁ブロック時の特殊即死判定と、戦闘による墓地送り時の世代交代誘発を実証する。
- **やること**:
  - 兵士 vs 防壁時の防壁表向き化、Joker/数字一致判定、および防壁の墓地送り処理の実装。
  - 戦闘墓地送り（`field -> grave`）時に、構成する Legacy Card から「世代交代」が正しく自動誘発する連携のテスト。

### Phase 14.5: CLI戦闘シナリオ統合
- **目的**: CLIシナリオランナー（`scenarioRunner.ts`）に戦闘シナリオを追加し、美しい戦闘ログを目視確認する。
- **やること**:
  - シナリオ7「アタック・ブロック・ダメージ判定の戦闘解決と世代交代の連鎖」を追加。
  - 割り込み魔法（アップ/ダウン）によるサイズ変動が戦闘結果に反映されることを美麗なログ出力とともに実証。
