# 新ルールYAML DSL 戦闘・解決モデル 移行計画 (Combat & Resolution Plan)

本ドキュメントは、新ルールYAML DSLにおいて「アタック」「ブロック」「ダメージ判定」を中心とする戦闘・解決モデルを安全に移植・構築するための具体的な移行計画および設計書です。

---

## 1. アタック・ブロック・ダメージ判定の基本方針

### 1.1. フェーズ概念の非導入とタイミング種別の活用
- **既存のフェーズ概念（アタックフェーズ、ブロックフェーズ等）は一切再導入しません**。
- 代わりに、新ルールYAML DSL上のアクション起動可能条件として、内部タイミング（`timing`）に以下の種別を導入して制御します。
  - `timing: main`: 通常の手番プレイヤーのメインアクション（アタック宣言等）。
  - `timing: block`: アタック解決後、防御側がブロッカーを能動的に指定してリクエストするタイミング。
  - `timing: damageJudge`: ブロック解決後、アタッカーとブロッカーの戦闘結果を評価するタイミング。
- これらはフェーズではなく、チャンス管理モデル（手番・チャンス所持）とステージ（LIFO）の制約下で実行可能なアクション条件として一貫して扱われます。

### 1.2. 1 vs 1 戦闘モデルの最小実装
- 移行検証の安全性を最優先し、本計画の初期段階では「1体のアタッカー vs 1体のブロッカー」による 1 vs 1 戦闘モデルに閉じて実装・実証を行います。
- 複数アタッカーの同時指定や、それに対する複数ブロックの複雑なバインディング拡張は、移行検証の成功後に段階的に行います。

### 1.3. 戦闘一時情報（`unit.battle`）の導入とクリーンアップ
- アクション解決間で戦闘状態（誰が誰を攻撃・ブロックしているか）を維持するため、ユニットオブジェクト内に一時戦闘情報 `battle` プロパティを導入します。
  - アタッカー: `unit.battle = { role: "attacker", targetPlayerKey: "p2" }`
  - ブロッカー: `unit.battle = { role: "blocker", blocksUnitId: "attackerUnitId" }`
- 戦闘プロセス（ダメージ判定）の完了時、または戦闘の中断（キャンセル）時には、`delete unit.battle` によって戦闘状態を完全に削除し、場や墓地のユニットに一切の一時情報が残らないクリーンな状態を維持します。

## 2. YAML DSLでの表現とアクション定義

各アクションは新YAML DSL（`tools/simulator/src/data/rules-vnext/examples/`）に定義され、宣言的な表現を持ちます。

### 2.1. アタックアクション (`action.attack`)
- タイミングは `timing: main`、速度は通常（`normal`）。
- 対象（`targets`）は、自分の場（`field`）にある攻撃ラベルを持つチャージ状態のキャラクター1体。

### 2.2. ブロックアクション (`action.block`)
- タイミングは `timing: block`、速度は通常（`normal`）。
- 対象は、自分の場にある防御ラベルを持つキャラクター1体。
- 誘発条件として、`action.attack` の正常解決イベント（`actionResolved`）を受けて誘発し、リクエストバッファ（`requestBuffer`）に積まれます。

### 2.3. ダメージ判定アクション (`action.damageJudge`)
- タイミングは `timing: damageJudge`、速度は通常（`normal`）。
- 対象をとらず、フィールド上のアタッカー・ブロッカーを自動スキャンしてサイズを比較します。
- 誘発条件として、`action.block` の正常解決イベントを受けて誘発し、リクエストバッファに積まれます。

## 3. 今後の実装フェーズ・ロードマップ

戦闘モデルの再現に向け、以下の段階に分割して実装を進めました。

### Phase 14.1: アタックアクションの最小実装 [完了]
- **目的**: アタッカーの指定、ドライブ状態への移行、およびアタッカーユニットへの戦闘一時情報の記録を実証する。
- **実施内容**:
  - `attack.yaml` (新YAML定義) の追加。
  - `ActionRequestValidator.ts` におけるバリデーションの実装（アタッカーが自分のフィールドに存在し、かつチャージ状態であることの検証）。
  - `commandHandlers.ts` に `startAttack` コマンドを実装し、アタッカーのドライブ状態移行および `battle` 情報の記録を完了。
  - 統合テスト `attack.test.ts` および CLI シナリオ7を追加し、挙動の整合性を実証。

### Phase 14.2: ブロックアクションの最小実装 [完了]
- **目的**: 防御側によるブロッカーの指定、およびブロッカーユニットへの戦闘一時情報の記録を実証する。
- **実施内容**:
  - `block.yaml` (新YAML定義) の追加。対象指定は `targetUnit` (ブロッカー) の1つのみ。
  - **`timing: block` 内部タイミングの導入**: ブロック起動可能条件を表す新タイミング種別を実装。これはブロックフェーズを意味するものではなく、チャンス管理下で実行可能条件のみを制御する内部タイミングです。
  - **自分を狙う相手のアタッカーの厳密な自動特定**: `declareBlock` コマンドにおいて、自分自身を攻撃している相手のアタッカーを厳密に特定してブロック対象とします。
  - **ブロッカー条件の最小実装**: 「`防御`ラベルを持つキャラクター」のみをブロック可能と規定。
  - ブロック解決時に `blocker.battle = { role: "blocker", blocksUnitId: attacker.unitId }` を記録し、ドライブ状態に移行。
  - `block.test.ts` 新規追加および `scenarioRunner.ts` (シナリオ7) のブロック対応への拡張を実施。

### Phase 14.3: ダメージ判定アクションの最小実装 (基本戦闘) [完了]
- **目的**: 兵士同士の基本的なサイズ比較による戦闘結果の解決を実証する。
- **実施内容**:
  - `damage-judge.yaml` のYAML定義の追加。タイミング `timing: damageJudge` を導入。
  - `ActionRequestValidator.ts` で、手番・チャンス所持・ステージ空、アタッカーおよび対応するブロッカーがそれぞれちょうど1体存在することを強制。
  - `AbilityEvaluator` に基づくサイズ比較を行い、敗北ユニットを墓地（`moveToGraveyard`）へ送り、戦闘一時情報 `battle` を `delete` して完全にクリーンアップ。
  - 墓地移動の際、構成カードごとに `cardMoved` イベントを個別に発行し、Legacy Card (J) 送りから既存の「世代交代」が自動誘発する連携を実証。
  - `damageJudge.test.ts` の追加、および `scenarioRunner.ts` (シナリオ7) へのダメージ判定プロセスの導入。

### Phase 14.4: 汎用誘発条件評価エンジンとリクエストバッファの最小実装 [完了]
- **目的**: 世代交代・ブロック・ダメージ判定を特別扱いせず、YAMLの `triggerCondition` 定義で統一評価し、公式用語「リクエストバッファ (`state.requestBuffer`)」へ蓄積する汎用エンジンを実証する。
- **実施内容**:
  - `TriggerResolver.ts` の新規追加。YAML アクションの `triggerCondition` 定義に基づき、イベント（`cardMoved`, `actionResolved`）発生時に誘発条件を型安全に評価。
  - 同一 `resolveTriggers` 呼び出し内に限定した 6-D-9 メイン系アクション排他判定（`isMainTriggeredRequest`）を実装し、2件目以降を `status: "discarded"` として理由付きで `history` に記録する安全弁を実装。
  - コントローラーの自動特定（ブロックは防御側、ダメージ判定はターンプレイヤー）を実装。
  - キャンセル時の誤誘発を完全に防止するため、`actionResolved` イベントの発行を `request.status === "resolved"` の時のみに厳密に制限。
  - 既存の即時解決処理と二重で動作（二重ライフめくり等）しないよう、今回のフェーズではリクエストバッファの自動消費・解決は行わない安全な最小実装にとどめ、既存テストおよび CLI フローへの影響ゼロを保証。
  - 網羅的な統合テスト `triggerResolver.test.ts` (ケース A〜J) を追加し、すべての安全弁がアサーションを含めて 100% 通ることを実証。

### Phase 14.5: CLI戦闘シナリオ統合 [完了]
- **目的**: シミュレーターの CLI シナリオランナーにアタック、ブロック、ダメージ判定、世代交代の連鎖、およびリクエストバッファの蓄積を追加し、一連の戦闘フローが完璧に動作することを目視確認する。
- **実施内容**:
  - `scenarioRunner.ts` のシナリオ7「アタックを宣言して戦闘状態を作り、ブロックで対応付ける」を拡張し、ダメージ判定解決までのフルプロセスを網羅。
  - アタックおよびブロック解決時に `[TRIGGER] ブロック が誘発しました` / `[TRIGGER] ダメージ判定 が誘発しました` の美麗なログを出力し、リクエストバッファへの蓄積と既存の手動実行フローが一切壊れず正常完走することを確認。
  - 割り込み魔法によるサイズ変動がダメージ判定結果に正しく反映される一連の連鎖挙動を目視確認。

## 6. 戦闘モデルの残された課題

本フェーズ（Phase 14.4）でのリクエストバッファ新設に伴い、以下の課題が将来のフェーズでの統合要件として整理されました。

1. **リクエストバッファの自動消費・自動解決の統合**:
   - 現在、リクエストバッファ（`state.requestBuffer`）への蓄積および 6-D-9 ルールによる破棄までは検証・アサートされていますが、バッファに積まれたリクエストの自動解決やチャンス移行処理との動的統合は行われていません。
   - 次フェーズ以降で、既存の即時イベント解決処理（`EffectInterpreter.dispatchEvent` 内での即時解決）と、バッファを消費して解決するフロー（コアフローの誘発処理）の統合を進め、二重実行されない安全な自動解決フローを実装する必要があります。
2. **ブロック可能条件・防壁の特殊判定の再現**:
   - ブロッカーに指定できる詳細な条件（一般兵と防壁の違い、カード数値によるブロック制約など）や、防壁特有の特殊判定（Joker 即死判定、数字一致判定など）を完全再現する必要があります。
3. **アタッカーの複数指定とブロック対応**:
   - 現在は「アタッカー1体 vs ブロッカー1体」の 1 vs 1 状況のみにバリデーション制限していますが、複数アタッカーおよび複数ブロックの対応関係バインドへの拡張が必要です。
4. **戦闘割り込み時のアタッカー状態変化の裁定**:
   - アタック宣言後、ダメージ判定解決までの間にアタッカーがツイスト等でチャージ状態に戻された場合の戦闘挙動（アタッカー資格剥奪か、それとも戦闘続行か）のルール厳密定義と整合を図る必要があります。
